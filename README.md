# HammerTime

YOLOv8-Nano tool detection framework. Supports a binary single-class classifier and a multi-class detector on a shared dataset. Both pipelines train from the same `data/` directory with a single shared `prepare_dataset.py`.

---

## Project Goal

Build an efficient, high-accuracy hammer detector. The purpose of this project is to deepen personal understanding of how data composition and process bottlenecks impact model performance.

The binary classifier treats all non-target images as background. The multi-class detector assigns explicit categories to visually similar tool types, applying Ashby's Law of Requisite Variety to stabilise the hammer decision boundary.

The strategies used to optimise hammer performance in the binary classifier are presumed to be transferable to generalised multi-class performance if replicated at the same depth across all classes. To maintain project focus, generalised poor performance among non-hammer classes is acceptable once outliers have been stabilised and diminishing returns on hammer optimisation have been reached.

---

## Classes

| ID | Class    | Role |
|----|----------|------|
| 0  | hammer   | Primary detection target |
| 1  | axe      | Boundary class, visual neighbour to hammer |
| 2  | wrench   | Boundary class, common co-occurrence with hammer |
| 3  | pickaxe  | Boundary class, handle/head confusion risk |

In binary mode, the target class is remapped to class 0 and all other labeled images are treated as background. Source label files are never modified.

---

## Success Criteria

Hammer class Precision exceeds V1 Run 9 baseline (0.8743) with Recall held above 0.85 on the same locked 93-image test set.

## Workflow

### Binary classifier (single target class)

```bash
python src/shared/prepare_dataset.py --target_class hammer
python src/binary/train_b.py
python src/binary/evaluate_b.py --split val
python src/binary/evaluate_b.py --split test --v1_compare
```

### Multi-class detector

```bash
python src/shared/prepare_dataset.py
python src/multi/train_m.py
python src/multi/evaluate_m.py --split val
python src/multi/evaluate_m.py --split test --v1_compare
```

---

## Key Design Decisions

**Immutable source labels.** `data/labels/` is never modified. Binary mode remaps labels in memory when writing to `dataset/`, so the same source data supports both pipelines without annotation duplication.

**Shared test manifest.** `dataset/test_manifest.txt` locks the test set on first run. New images always route to train/val only. The same 93-image test set is used for both the binary and multi-class capstone comparison.

**Switching modes.** Re-run `prepare_dataset.py` with or without `--target_class` to rebuild `dataset/` for either pipeline. Runs are stored in `runs/binary/` and `runs/multi/` separately.

## Scripts Reference

### Shared

| Script | Purpose |
|--------|---------|
| `src/shared/prepare_dataset.py` | Validate, deduplicate, split 70/15/15, lock test set. Use `--target_class` for binary mode. |
| `src/shared/count_dataset.py` | Dataset composition with per-class annotation counts |

### Binary

| Script | Purpose |
|--------|---------|
| `src/binary/train_b.py` | Train binary classifier. Outputs to `runs/binary/` |
| `src/binary/evaluate_b.py` | Val/test evaluation with optional V1 baseline comparison |
| `src/binary/predict_b.py` | Inference |
| `src/binary/analyze_failures_b.py` | False negative analysis |
| `src/binary/analyze_false_positives_b.py` | False positive analysis |
| `src/binary/export_b.py` | Export to ONNX/TFLite/CoreML |

### Multi

| Script | Purpose |
|--------|---------|
| `src/multi/train_m.py` | Train multi-class detector. Outputs to `runs/multi/` |
| `src/multi/evaluate_m.py` | Val/test evaluation with per-class breakdown and V1 comparison |
| `src/multi/predict_m.py` | Inference with per-class output |
| `src/multi/analyze_failures_m.py` | False negative analysis, filterable by class |
| `src/multi/analyze_false_positives_m.py` | False positive analysis, filterable by class |
| `src/multi/export_m.py` | Export to ONNX/TFLite/CoreML |

## Known Issues

- YOLOv8 auto-increments run names (train-2, train-3 etc.). Update `--weights` path accordingly.
- macOS creates " 2" duplicate files on filename collision. Run `find data/classes -name "* 2.*" -delete` before `prepare_dataset.py`.
- `dataset/` is fully disposable and must be rebuilt when switching between binary and multi-class modes.
